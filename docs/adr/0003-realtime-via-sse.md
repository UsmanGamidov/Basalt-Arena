# ADR 0003. Realtime через SSE, а не WebSocket

- **Статус:** принято
- **Дата:** 2026-06

## Контекст

Нужны живые обновления зала славы, лидерборда и статусов отправок. Связь
односторонняя: сервер → клиент («данные изменились»). Двусторонний канал не нужен —
действия пользователя идут обычными REST-запросами.

## Решение

Используем **Server-Sent Events** (`GET /v2/events`):

- Доменные сервисы публикуют событие через `RealtimeService` после изменений.
- Эндпоинт транслирует события (heartbeat раз в 25с держит соединение).
- Тело события не содержит PII — только сигнал; клиент перезапрашивает данные
  через авторизованные эндпоинты и обновляет общую шину live-refresh.
- На клиенте интервальный поллинг остаётся фолбэком, если SSE недоступен.

WebSocket (Socket.io) отклонён: для односторонних уведомлений избыточен, тянет
свой транспорт и при масштабировании — Redis-adapter (см. ADR 0002).

## Последствия

- Работает на одном сервисе Render без брокера; EventSource сам переподключается.
- `compression` настроен пропускать `text/event-stream`, иначе буфер ломает доставку.
- Для нескольких инстансов понадобится общая шина (Redis pub/sub) — точка пересмотра.
